# DESIGN

## Модель данных

### `approval_requests`

- основная заявка на согласование;
- хранит `workspace_id`, источник (`source_type`, `source_id`), контентные поля, список ревьюеров, текущий статус;
- финальные статусы: `approved`, `rejected`, `canceled`.

### `approval_events`

- аудит-лог всех успешных изменений;
- фиксирует `actor_user_id`, действие (`created/approved/rejected/canceled`), comment/reason и timestamp.

### `idempotency_records`

- обеспечивает идемпотентность создания;
- уникальная пара `(workspace_id, idempotency_key)` связывается с созданной заявкой и hash payload.

### `outbox_events`

- таблица outbox для последующей интеграции с брокером событий;
- для каждого успешного изменения создается событие `ApprovalRequest*`.

## Границы сервиса

- Сервис не реализует соседние доменные сущности (publication/scenario/edit/external), работает только с их ID.
- Изоляция данных обеспечивается обязательным `workspace_id` в path и проверкой auth-заглушки.

## Обработка повторов

- Если для `POST /approval-requests` передан `X-Idempotency-Key`:
  - первый запрос создает запись;
  - повтор с тем же payload возвращает уже созданную заявку;
  - повтор с другим payload и тем же ключом возвращает `409`.

## События и интеграции

- Использован outbox-подход: доменные изменения и запись outbox создаются в одной транзакции.
- Это позволяет безопасно добавить отдельный publisher в будущем без потери событий.

## Известные компромиссы

- Auth реализован как заглушка через заголовки, без JWT/OAuth.
- Список `reviewerUserIds` хранится как JSON-строка в одной колонке для кросс-БД простоты.
- Конкурентные записи в SQLite ограничены возможностями локальной БД; в production ожидается PostgreSQL.
